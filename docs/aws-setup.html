2 · Enable Bedrock model access console

All content generation models go in one region (eu-west-1 recommended).

eu-central-1 (Frankfurt) has none of the content models — not Claude Haiku, not Nova text, not Nova Canvas. So enable them all in eu-west-1 (the region you already use for images). Embeddings (Titan) and Polly stay in eu-central-1, where they work.
AWS console · Bedrock · eu-west-1

Request access in Ireland (eu-west-1) for all three

Switch region to eu-west-1 (top-right) → Model access → enable:

• Anthropic — Claude 3 Haiku (post / video-script text)
• Amazon — Nova text model (topic extraction)
• Amazon — Nova Canvas (images)

If eu-west-1 happens not to offer Haiku/Nova in your account, pick a region that does (e.g. us-east-1) and set BEDROCK_TEXT_REGION / BEDROCK_IMAGE_REGION to it in section 1.

3 · Content bucket + CloudFront console / CLI

Where PULSE images and SIGNAL videos live and are served from.

On EC2 (or console)

Create a private S3 bucket

aws s3 mb s3://personal-intelligence-content --region eu-central-1

Keep “Block all public access” ON — CloudFront will serve it.

AWS console · CloudFront

Create a CloudFront distribution for it

New distribution → Origin = the bucket above → Origin access control (OAC) (so the bucket stays private) → create. Copy the policy CloudFront offers and paste it onto the bucket. Use a separate distribution from the dashboard one (avoids the 5-behaviour limit).

Take the distribution domain (e.g. d1abcd.cloudfront.net) and set it as CLOUDFRONT_BASE_URL=https://d1abcd.cloudfront.net in .env.

4 · IAM permissions for the EC2 role console / CLI

The server needs to call Bedrock, Polly, and write to the content bucket.

AWS console · IAM · the EC2 instance role

Attach this inline policy

IAM → Roles → the role your EC2 instance uses → Add permissions → Create inline policy → JSON:

{
  "Version": "2012-10-17",
  "Statement": [
    { "Sid": "Bedrock", "Effect": "Allow", "Action": "bedrock:InvokeModel", "Resource": "*" },
    { "Sid": "Polly",   "Effect": "Allow", "Action": "polly:SynthesizeSpeech", "Resource": "*" },
    { "Sid": "Content", "Effect": "Allow",
      "Action": ["s3:PutObject","s3:GetObject"],
      "Resource": "arn:aws:s3:::personal-intelligence-content/*" }
  ]
}

(Bedrock is scoped to * on purpose — cross-region inference-profile ARNs are fiddly; tighten later if you like.)

7 · Restart & verify on EC2

Load the new config and confirm the agents can read memory.

On EC2

Anywhere (use a real dashboard token)

Verify the agents can reach memory

Grab a token in the browser console on the dashboard: copy(sessionStorage.getItem('pi_access_token')), then:

TOK='<paste real token>'
curl -s -o /dev/null -w "health: %{http_code}\n" -H "Authorization: Bearer $TOK" \
  https://adel-intelligence.com/api/health/daily/$(date +%F)

200 = the health & finance agents work. A 503 mentioning INTERNAL_SECRET means it isn't set/loaded — recheck section 1 and restart.

8 · Turn on content generation (Phase 8)

First test by hand, then schedule it. PULSE/SIGNAL build content from your brain's memories, so make sure you've written/uploaded a few first.

On EC2

Smoke test — run the pipeline by hand (in order)

Topics must exist before posts/videos. These use INTERNAL_SECRET, not a user token.

S=$(grep '^INTERNAL_SECRET=' ~/personal-intelligence/.env | cut -d= -f2-)
curl -s -XPOST localhost:8000/api/content/jobs/topics/run -H "X-Internal-Secret: $S"; sleep 45
curl -s -XPOST localhost:8000/api/content/jobs/posts/run  -H "X-Internal-Secret: $S" \
     -H 'Content-Type: application/json' -d '{"count":3}'; sleep 60
curl -s -XPOST localhost:8000/api/content/jobs/videos/run -H "X-Internal-Secret: $S" \
     -H 'Content-Type: application/json' -d '{"count":1}'
journalctl -u personal-intelligence-api -n 60 --no-pager   # watch for errors

Then check the PULSE and SIGNAL pages on the dashboard, and that objects appeared in the S3 content bucket.

Your call — tell me which and I'll generate it

Make it run on a schedule

Once the smoke test produces content, pick one and I'll build the artifact for you:

• EC2 systemd timer (simplest — like the existing email-sync cron): topics daily, posts twice a day, a video daily. Version-controlled so deploys install it.
• EventBridge Scheduler → API destination (serverless): the AWS-native option.

If a call returns HTML instead of JSON (“unexpected character at line 1”): the API request is hitting a 404/error that CloudFront rewrites to the SPA — check the URL/route, not the data.
